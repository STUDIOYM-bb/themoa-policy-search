# 정책 데이터 수집 기능 설계서

## 1. 기능 개요

정책 데이터 수집 기능은 여러 외부 정책 데이터 출처에서 정책 정보를 가져와 서비스 내부 데이터베이스에 저장하고, 검색 및 추천 기능에서 사용할 수 있도록 정제하는 기능이다.

우리 서비스에서 수집하는 정책 데이터 출처는 다음 네 가지다.

- 온통청년
- 행정안전부 공공서비스 정보
- 지자체 복지서비스
- 중앙부처 복지서비스

정책 검색 기능은 사용자가 자연어로 입력한 조건과 정책 데이터를 비교해 신청 가능성을 판단해야 한다. 따라서 외부 API를 사용자 검색 시점마다 직접 호출하는 방식보다는, 정책 데이터를 미리 수집해 내부 DB에 저장하고, 검색 기능은 내부 DB의 통합 정책 데이터를 기준으로 동작하도록 설계한다.

전체 흐름은 다음과 같다.

```text
외부 정책 데이터 출처
↓
출처별 정책 데이터 수집
↓
원본 데이터 저장
↓
공통 정책 구조로 정제
↓
중복 정책 확인
↓
신규 정책 저장 또는 기존 정책 갱신
↓
검색/추천/캘린더/알림 기능에서 활용
```

---

## 2. 수집 기능의 목적

정책 데이터 수집 기능의 목적은 단순히 외부 API 데이터를 가져오는 것이 아니다.

출처마다 데이터 구조, 필드명, 지역 표현, 신청 기간 표현, 지원 대상 표현이 다를 수 있기 때문에, 여러 출처의 정책 데이터를 하나의 서비스 기준으로 정리하는 것이 핵심이다.

이 기능을 통해 서비스는 다음을 달성한다.

- 여러 출처의 정책 정보를 한 번에 관리한다.
- 정책 검색 기능이 외부 API 장애에 직접 영향을 받지 않도록 한다.
- 서로 다른 출처의 정책 데이터를 공통 구조로 변환한다.
- 중복 정책이 반복 노출되지 않도록 관리한다.
- 정책 신청 기간을 기준으로 캘린더와 알림 기능에 연동한다.
- 자연어 검색과 신청 가능성 판별에 필요한 조건 데이터를 확보한다.

## 3. 정책 데이터 출처

### 3.1 온통청년

온통청년은 청년정책 중심의 데이터를 수집하는 출처다.

주요 수집 대상은 다음과 같다.

- 청년 취업 정책
- 청년 주거 정책
- 청년 금융 정책
- 청년 교육 정책
- 청년 창업 정책
- 청년 복지 정책
- 청년 생활지원 정책

온통청년 데이터는 특히 사용자가 “청년”, “대학생”, “취준생”, “무직 청년”, “월세 지원”, “청년 적금” 등으로 검색했을 때 활용도가 높다.

---

### 3.2 행정안전부 공공서비스 정보

행정안전부 공공서비스 정보는 정부 및 공공기관에서 제공하는 공공서비스성 정책 데이터를 수집하는 출처다.

주요 수집 대상은 다음과 같다.

- 정부 공공서비스
- 행정 지원 서비스
- 국민 대상 지원사업
- 생활 지원 서비스
- 민원/신청성 공공서비스
- 기관별 공공 지원 정보

이 데이터는 청년정책에 한정되지 않고, 더 넓은 범위의 공공서비스를 검색 결과에 포함하기 위해 사용한다.

---

### 3.3 지자체 복지서비스

지자체 복지서비스는 지방자치단체에서 제공하는 지역 단위 복지정책을 수집하는 출처다.

주요 수집 대상은 다음과 같다.

- 시/도 단위 복지서비스
- 시/군/구 단위 복지서비스
- 지역 청년 지원
- 지역 저소득층 지원
- 지역 주거 지원
- 지역 생활안정 지원
- 지역 취약계층 지원
- 지역별 현금성 지원 정책

이 데이터는 사용자가 “수원”, “경기도”, “서울”, “우리 동네”, “지역 지원금”처럼 지역 중심으로 검색했을 때 중요하게 사용된다.

---

### 3.4 중앙부처 복지서비스

중앙부처 복지서비스는 중앙정부 부처에서 제공하는 전국 단위 복지정책을 수집하는 출처다.

주요 수집 대상은 다음과 같다.

- 전국 단위 복지서비스
- 중앙정부 생활지원 정책
- 고용/복지/보건 관련 정책
- 저소득층 지원
- 청년 대상 중앙정부 정책
- 가족, 장애인, 노인, 아동 등 대상별 복지서비스

이 데이터는 특정 지역에 국한되지 않는 전국 단위 정책을 검색 결과에 포함하기 위해 사용한다.

---

## 4. 수집 방식

정책 데이터는 사용자가 검색할 때마다 외부 API를 직접 호출하지 않는다.

대신 서버가 외부 API를 호출하여 정책 데이터를 내부 DB에 저장하고, 사용자의 검색 요청은 내부 DB를 기준으로 처리한다.

이 방식을 사용하는 이유는 다음과 같다.

- 검색 속도를 안정적으로 유지할 수 있다.
- 외부 API 장애가 발생해도 기존에 저장된 정책 데이터로 검색을 유지할 수 있다.
- API 호출 제한이나 트래픽 초과 문제를 줄일 수 있다.
- 출처마다 다른 데이터 구조를 미리 정제할 수 있다.
- 신청 가능성 판별에 필요한 조건 데이터를 별도로 가공할 수 있다.
- 관심 정책 저장, 캘린더, 알림 기능과 쉽게 연동할 수 있다.
- 여러 출처에서 같은 정책이 수집되더라도 중복을 관리할 수 있다.

---

## 5. 전체 수집 흐름

정책 데이터 수집은 다음 흐름으로 진행된다.

```text
1. 수집 작업 실행
2. 수집 대상 출처 선택
3. 출처별 API 인증 정보 확인
4. 출처별 API 호출
5. 페이지 단위로 정책 목록 조회
6. API 응답 데이터 파싱
7. 원본 데이터 저장
8. 출처별 응답 데이터를 공통 정책 구조로 변환
9. 지역, 대상, 분야, 날짜, 키워드 정제
10. 중복 정책 여부 확인
11. 신규 정책 저장 또는 기존 정책 갱신
12. 정책 신청 상태 계산
13. 수집 결과 로그 저장
14. 검색/추천에 사용할 정책 데이터 갱신
```

## 6. 수집 기능의 책임 범위

정책 수집 기능은 외부 데이터를 가져와 내부에서 사용할 수 있는 정책 데이터로 만드는 역할을 담당한다.

수집 기능이 담당하는 내용은 다음과 같다.

- 외부 API 인증 정보 관리
- 출처별 API 호출
- 페이지네이션 처리
- API 응답 파싱
- 원본 데이터 저장
- 공통 정책 데이터 구조로 변환
- 지역 정제
- 대상 정제
- 분야 정제
- 날짜 정제
- 검색 키워드 생성
- 중복 정책 판단
- 신규 정책 저장
- 기존 정책 갱신
- 신청 상태 계산
- 수집 성공/실패 로그 저장
- 수동 수집 실행
- 자동 수집 스케줄러 실행

반대로 수집 기능이 담당하지 않는 내용은 다음과 같다.

- 사용자의 자연어 검색어 분석
- 검색 조건 추출
- 사용자 조건과 정책 조건 비교
- 신청 가능성 판별
- 부족 조건 질문 생성
- 정책 결과 카드 구성
- 사용자별 관심 정책 저장

위 내용은 정책 검색 및 추천 기능에서 처리한다.

## 7. 출처별 수집기 구조

출처마다 API 요청 방식, 응답 필드, 데이터 구조가 다르기 때문에 출처별 수집기를 분리한다.

예상 구조는 다음과 같다.

```text
PolicyCollector
├─ YouthCenterPolicyCollector
├─ GovServiceInfoCollector
├─ LocalWelfareServiceCollector
└─ CentralWelfareServiceCollector
```

각 수집기는 다음 역할을 담당한다.

```text
1. 해당 출처의 API 호출
2. 해당 출처의 응답 파싱
3. 해당 출처의 원본 데이터 저장
4. 해당 출처 데이터를 공통 정책 데이터로 변환
```

출처별 수집기가 정책 데이터를 직접 최종 저장하지 않고, 공통 정책 저장 서비스로 넘기는 구조를 사용한다.

```text
출처별 Collector
↓
공통 정책 DTO 생성
↓
PolicyCollectionService
↓
중복 판단
↓
정책 저장 또는 갱신
↓
수집 로그 저장
```

이렇게 하면 출처가 추가되더라도 검색 기능이나 저장 로직 전체를 크게 바꾸지 않고 확장할 수 있다.

---

## 8. 공통 정책 데이터 구조

서로 다른 출처에서 수집한 데이터를 하나의 검색 기능에서 사용하기 위해 공통 정책 데이터 구조로 정제한다.

공통 정책 데이터는 다음 정보를 포함한다.

| 항목               | 설명                                                                         |
| ------------------ | ---------------------------------------------------------------------------- |
| 데이터 출처        | 온통청년, 행정안전부 공공서비스 정보, 지자체 복지서비스, 중앙부처 복지서비스 |
| 출처별 정책 식별값 | 외부 출처에서 제공하는 정책 고유값                                           |
| 정책명             | 정책 또는 서비스 이름                                                        |
| 주관 기관          | 정책을 운영하는 기관, 부처, 지자체                                           |
| 지역               | 정책 대상 지역                                                               |
| 지역 코드          | 지역을 식별하기 위한 코드                                                    |
| 분야               | 주거, 취업, 금융, 교육, 복지 등 내부 서비스 기준 분야                        |
| 지원 대상          | 청년, 대학생, 구직자, 저소득층 등                                            |
| 연령 조건          | 신청 가능한 최소/최대 나이                                                   |
| 취업 상태 조건     | 재직, 미취업, 구직 중 등                                                     |
| 학생 여부 조건     | 대학생, 졸업생, 휴학생 등                                                    |
| 소득 조건          | 기준 중위소득, 소득 유무 등                                                  |
| 주거 조건          | 무주택, 월세, 전세 등                                                        |
| 가구 조건          | 1인 가구, 신혼부부, 한부모 가구 등                                           |
| 선정 기준          | 정책 대상자를 선정하는 상세 기준                                             |
| 지원 내용          | 금액, 서비스, 교육, 상담 등 실제 지원 내용                                   |
| 신청 방법          | 온라인 신청, 방문 신청 등                                                    |
| 신청 시작일        | 정책 신청 시작일                                                             |
| 신청 마감일        | 정책 신청 마감일                                                             |
| 신청 상태          | 신청 가능, 신청 예정, 신청 마감, 상시 신청 등                                |
| 공식 링크          | 정책 상세 페이지 또는 신청 URL                                               |
| 문의처             | 담당 기관 또는 문의 전화                                                     |
| 검색 키워드        | 검색 정확도를 높이기 위한 키워드                                             |
| 원본 데이터 참조   | 원본 API 응답과 연결하기 위한 값                                             |
| 마지막 수집 일시   | 정책 데이터가 마지막으로 수집 또는 갱신된 시간                               |

## 9. 원본 데이터와 정제 데이터 분리

정책 데이터는 원본 데이터와 정제 데이터를 분리해서 관리한다.

### 9.1 원본 데이터

원본 데이터는 외부 API에서 받은 응답을 가능한 한 그대로 저장한 데이터다.

원본 데이터를 저장하는 이유는 다음과 같다.

- API 응답 구조가 변경되었을 때 비교할 수 있다.
- 정제 과정에서 누락된 데이터를 다시 처리할 수 있다.
- 데이터 오류나 수집 실패 원인을 확인할 수 있다.
- 새로운 필드를 추가할 때 원본 데이터를 기준으로 재가공할 수 있다.
- 출처별 데이터 품질을 비교할 수 있다.

원본 데이터에는 다음 정보가 포함될 수 있다.

| 항목               | 설명                              |
| ------------------ | --------------------------------- |
| 데이터 출처        | 어느 출처에서 가져온 데이터인지   |
| 출처별 정책 식별값 | 외부 API에서 제공하는 정책 식별값 |
| 요청 URL           | API 호출 URL                      |
| 요청 파라미터      | API 호출 시 사용한 파라미터       |
| 응답 원문          | XML, JSON 등 API 응답 원문        |
| 파싱 상태          | 파싱 성공 또는 실패 여부          |
| 수집 일시          | 데이터를 수집한 시간              |
| 오류 메시지        | 실패 시 오류 내용                 |

---

### 9.2 정제 데이터

정제 데이터는 서비스에서 실제로 사용하는 정책 데이터다.

정제 데이터는 다음 기능에서 사용된다.

- 자연어 정책 검색
- 정책 목록 조회
- 정책 상세 조회
- 신청 가능성 판별
- 관심 정책 저장
- 캘린더 표시
- 알림 기능

정제 데이터는 원본 데이터의 내용을 서비스 기준에 맞게 변환한 결과다.

예를 들어 출처마다 정책 분야 표현이 다를 수 있다.

```text
온통청년: 주거
행정안전부 공공서비스 정보: 생활안정
지자체 복지서비스: 주거급여/주택지원
중앙부처 복지서비스: 주거지원
```

서비스에서는 위 데이터를 내부 카테고리인 `주거`로 통일하여 저장한다.

---

## 10. 데이터 정제 기준

외부 API 데이터는 바로 검색과 추천에 사용하기 어렵다.

따라서 수집 후 다음 기준에 따라 정제한다.

---

### 10.1 지역 정제

정책 지역 정보는 서비스 내 표준 지역명으로 정리한다.

예시:

```text
경기 → 경기도
수원 → 경기도 수원시
수원시 → 경기도 수원시
서울 → 서울특별시
서울시 → 서울특별시
전국 → 전국
중앙부처 정책 → 전국
```

정책 지역이 전국 단위인 경우 `전국`으로 저장한다.

지자체 정책의 경우 가능한 범위 내에서 시/도, 시/군/구 단위까지 저장한다.

---

### 10.2 대상 정제

정책 대상은 검색과 추천에 사용할 수 있도록 표준화한다.

예시:

```text
청년
대학생
구직자
미취업자
재직자
신혼부부
창업자
프리랜서
저소득층
장애인
노인
아동
가족
여성
소상공인
```

정책 설명에 여러 대상이 포함된 경우 복수 대상값으로 저장할 수 있다.

---

### 10.3 분야 정제

정책 분야는 서비스 내 카테고리로 변환한다.

예시:

```text
일자리
주거
교육
복지
금융
창업
문화
생활지원
건강
돌봄
```

출처별 분야명이 다르더라도 사용자가 보기 쉬운 내부 카테고리로 통일한다.

---

### 10.4 날짜 정제

신청 시작일과 마감일은 날짜 타입으로 변환한다.

날짜 정보가 텍스트로 제공되는 경우 가능한 범위 내에서 파싱한다.

예시:

```text
2026.07.01 ~ 2026.07.31
→ 신청 시작일: 2026-07-01
→ 신청 마감일: 2026-07-31
```

날짜 파싱이 불가능한 경우에는 원문은 유지하고, 시작일과 마감일은 비워둘 수 있다.

신청 기간이 없는 정책은 `확인 필요` 또는 `상시 신청`으로 분류한다.

---

### 10.5 키워드 정제

정책명, 정책 설명, 지원 내용, 선정 기준, 신청 방법에서 검색에 사용할 키워드를 추출한다.

예시:

```text
청년 월세 지원
→ 청년, 월세, 주거, 지원금, 임차료
```

키워드는 자연어 검색 결과의 정확도를 높이기 위해 사용한다.

---

### 10.6 조건 문장 정제

정책 신청 조건은 긴 문장으로 제공될 수 있다.

예시:

```text
만 19세 이상 34세 이하의 무주택 청년으로서 기준 중위소득 150% 이하인 자
```

가능한 경우 다음과 같이 분리한다.

```text
최소 나이: 19
최대 나이: 34
대상: 청년
주거 조건: 무주택
소득 조건: 기준 중위소득 150% 이하
```

단, 자동 분리가 어려운 경우 원문 조건을 유지하고 검색 결과에서는 `확인 필요`로 처리한다.

---

## 11. 중복 정책 처리

여러 출처에서 같은 정책 또는 유사한 정책이 중복 수집될 수 있다.

예를 들어 같은 청년 월세 지원 정책이 온통청년과 지자체 복지서비스 양쪽에서 수집될 수 있다.

이 경우 같은 정책이 검색 결과에 반복 노출되지 않도록 중복 판단이 필요하다.

---

### 11.1 같은 출처 안에서의 중복 판단

같은 출처 안에서는 다음 기준으로 중복을 판단한다.

```text
1. 출처별 정책 식별값이 동일한 경우 같은 정책으로 판단
2. 출처별 정책 식별값이 없는 경우 정책명 + 기관명 + 지역 기준으로 판단
3. 신청 기간이 다른 경우 같은 정책의 다른 회차인지 확인
```

---

### 11.2 서로 다른 출처 간 중복 판단

서로 다른 출처에서 같은 정책이 수집될 수 있으므로 다음 기준으로 유사 정책 여부를 판단한다.

```text
1. 정책명 유사도
2. 주관 기관 또는 소관 기관 유사도
3. 대상 지역 일치 여부
4. 신청 기간 일치 여부
5. 공식 링크 일치 여부
6. 지원 내용 유사도
```

---

### 11.3 중복 처리 방식

중복 처리는 보수적으로 진행한다.

정책명은 비슷해도 지역, 기관, 기간, 대상이 다르면 다른 정책일 수 있기 때문이다.

| 상황                                  | 처리 방식                                      |
| ------------------------------------- | ---------------------------------------------- |
| 완전히 동일한 정책                    | 하나의 대표 정책으로 묶고 출처 목록만 추가     |
| 같은 정책으로 보이나 일부 정보가 다름 | 대표 정책을 유지하고 출처별 원본 데이터는 보존 |
| 이름은 같지만 기관/지역/기간이 다름   | 별도 정책으로 저장                             |
| 같은 정책의 다른 회차                 | 별도 회차 또는 별도 정책으로 저장              |
| 판단이 애매함                         | 자동 병합하지 않고 별도 정책으로 저장          |

---

## 12. 정책 신청 상태 관리

정책은 신청 기간과 데이터 상태에 따라 신청 상태를 가진다.

| 상태      | 설명                                   |
| --------- | -------------------------------------- |
| 신청 가능 | 현재 신청 기간에 해당하는 정책         |
| 신청 예정 | 아직 신청 시작 전인 정책               |
| 신청 마감 | 신청 마감일이 지난 정책                |
| 상시 신청 | 특정 마감일 없이 상시 신청 가능한 정책 |
| 확인 필요 | 신청 기간 정보가 부족한 정책           |
| 비활성    | 사용자에게 노출하지 않을 정책          |

정책 신청 상태는 신청 시작일, 신청 마감일, 상시 신청 여부, 현재 날짜를 기준으로 자동 계산한다.

신청 기간 정보가 없는 정책은 캘린더에 표시하지 않거나, 상세 페이지에서 `확인 필요`로 안내한다.

---

## 13. 수집 실행 방식

정책 수집은 수동 실행과 자동 실행을 모두 고려한다.

### 13.1 수동 수집

수동 수집은 관리자 또는 개발자가 직접 정책 데이터를 수집하는 방식이다.

개발 초기에는 수집 결과를 바로 확인해야 하므로 수동 수집을 먼저 구현하는 것이 좋다.

수동 수집은 다음 방식으로 나눌 수 있다.

- 전체 출처 수동 수집
- 온통청년만 수동 수집
- 행정안전부 공공서비스 정보만 수동 수집
- 지자체 복지서비스만 수동 수집
- 중앙부처 복지서비스만 수동 수집
- 실패한 출처만 재수집

---

### 13.2 자동 수집

자동 수집은 스케줄러가 정해진 시간에 정책 데이터를 가져오는 방식이다.

초기 운영 기준은 하루 1회 자동 수집으로 한다.

예시:

```text
매일 새벽 03:00 정책 데이터 자동 수집
```

출처별 장애가 전체 수집 실패로 이어지지 않도록, 출처별 수집 작업은 독립적으로 실행한다.

예를 들어 온통청년 수집이 실패하더라도 행정안전부 공공서비스 정보, 지자체 복지서비스, 중앙부처 복지서비스 수집은 계속 진행한다.

---

## 14. API 인증 정보 관리

각 출처별 API 인증 정보는 코드에 직접 작성하지 않는다.

환경변수 또는 별도 설정 파일로 관리한다.

예시:

```properties
YOUTH_CENTER_API_KEY=발급받은_온통청년_API_키
GOV_SERVICE_API_KEY=발급받은_공공서비스_API_키
LOCAL_WELFARE_API_KEY=발급받은_지자체복지서비스_API_키
CENTRAL_WELFARE_API_KEY=발급받은_중앙부처복지서비스_API_키
```

Spring Boot 설정 예시:

```yaml
policy:
  collection:
    youth-center:
      api-key: ${YOUTH_CENTER_API_KEY}
    gov-service-info:
      api-key: ${GOV_SERVICE_API_KEY}
    local-welfare:
      api-key: ${LOCAL_WELFARE_API_KEY}
    central-welfare:
      api-key: ${CENTRAL_WELFARE_API_KEY}
```

API 인증 정보는 GitHub에 업로드하지 않는다.

`.env`, `application-secret.yml`, 서버 환경변수 등을 사용하여 관리한다.

---

## 15. 페이지 단위 수집 기준

정책 데이터는 한 번에 모두 조회하지 않고 페이지 단위로 수집한다.

예시:

```text
page=1, size=100
page=2, size=100
page=3, size=100
...
```

출처마다 페이지 파라미터 이름이 다를 수 있으므로, 출처별 수집기에서 해당 출처에 맞는 파라미터로 변환한다.

처리 기준은 다음과 같다.

```text
1. 첫 페이지부터 수집한다.
2. 응답 데이터가 없을 때까지 다음 페이지를 조회한다.
3. 전체 건수 정보가 제공되는 경우 전체 페이지 수를 계산하여 수집한다.
4. 특정 페이지에서 실패하더라도 전체 수집을 중단하지 않는다.
5. 실패한 페이지는 수집 로그에 기록한다.
6. 실패한 페이지는 재수집 대상에 포함한다.
```

---

## 16. 수집 실패 처리

외부 API 호출 또는 데이터 저장 과정에서 실패가 발생할 수 있다.

실패 상황은 다음과 같다.

| 실패 상황      | 설명                                  |
| -------------- | ------------------------------------- |
| API 인증 실패  | 인증키 오류 또는 만료                 |
| API 응답 실패  | 외부 API 서버 오류                    |
| 응답 파싱 실패 | XML/JSON 구조 변경 또는 깨진 데이터   |
| 필수값 누락    | 정책명, 기관명 등 주요 데이터 없음    |
| DB 저장 실패   | 데이터 타입 오류, 중복 제약 조건 오류 |
| 네트워크 오류  | 타임아웃, 연결 실패                   |
| 출처별 장애    | 특정 출처 API만 실패                  |

실패 처리 방식은 다음과 같다.

```text
1. 실패한 요청 정보를 로그로 저장한다.
2. 실패한 출처, 페이지, 정책 정보를 기록한다.
3. 특정 출처가 실패해도 다른 출처 수집은 계속 진행한다.
4. 전체 수집 작업을 중단하지 않고 가능한 데이터는 저장한다.
5. 필요 시 실패한 출처 또는 실패한 페이지 단위로 재수집한다.
```

---

## 17. 수집 로그 관리

정책 수집 작업이 실행될 때마다 수집 로그를 저장한다.

수집 로그에는 다음 정보가 포함될 수 있다.

| 항목             | 설명                                |
| ---------------- | ----------------------------------- |
| 수집 출처        | 어떤 출처를 수집했는지              |
| 수집 시작 시간   | 수집이 시작된 시간                  |
| 수집 종료 시간   | 수집이 종료된 시간                  |
| 수집 상태        | 성공, 일부 실패, 실패               |
| API 요청 횟수    | 외부 API를 호출한 횟수              |
| 전체 수집 건수   | API에서 가져온 전체 데이터 수       |
| 신규 저장 건수   | 새로 저장된 정책 수                 |
| 갱신 건수        | 기존 정책 중 갱신된 정책 수         |
| 제외 건수        | 중복 또는 필수값 누락으로 제외된 수 |
| 실패 건수        | 저장 또는 파싱 실패 건수            |
| 대표 오류 메시지 | 실패 시 대표 오류 내용              |
| 실행 방식        | 자동 수집 또는 수동 수집            |

---

## 18. 관리자용 수집 기능

정책 데이터는 자동 수집뿐 아니라 관리자가 직접 갱신할 수 있어야 한다.

관리자 기능은 다음과 같이 확장할 수 있다.

- 전체 정책 수동 수집
- 출처별 정책 수동 수집
- 수집 결과 확인
- 실패 데이터 재수집
- 정책 비활성화
- 정책 정보 수정
- 출처별 마지막 수집 성공/실패 확인

초기 구현에서는 관리자 화면까지 만들지 않더라도, 개발자 또는 관리자가 호출할 수 있는 수동 수집 API는 제공하는 것이 좋다.

## 19. 구현 범위

초기 구현에서는 모든 자동화와 정제 로직을 완벽하게 구현하기보다, 다음 범위를 우선 구현한다.

### 19.1 1차 구현 범위

```text
1. 4개 출처 API 인증 정보 설정
2. 출처별 수집기 클래스 분리
3. 온통청년 정책 데이터 수집
4. 행정안전부 공공서비스 정보 수집
5. 지자체 복지서비스 데이터 수집
6. 중앙부처 복지서비스 데이터 수집
7. 페이지 단위 데이터 수집
8. API 응답 파싱
9. 정책명, 기관, 지역, 분야, 지원 내용, 신청 기간, 공식 링크 저장
10. 데이터 출처 저장
11. 기존 정책 중복 확인 후 저장 또는 갱신
12. 전체 수동 수집 실행
13. 출처별 수동 수집 실행
14. 수집 결과 로그 출력
```

### 19.2 2차 구현 범위

```text
1. 매일 자동 수집 스케줄러 적용
2. 원본 데이터 저장 테이블 추가
3. 수집 실패 로그 테이블 추가
4. 지역/대상/분야 정규화 고도화
5. 정책 신청 상태 자동 계산
6. 검색 키워드 자동 생성
7. 출처 간 중복 정책 그룹화
8. 실패한 출처 또는 페이지 재수집 기능
```

### 19.3 확장 구현 범위

## 20. 검색 기능과의 관계

정책 수집 기능은 자연어 검색 기능의 기반 데이터 역할을 한다.

자연어 검색 기능은 외부 API를 직접 검색하는 것이 아니라, 4개 출처에서 수집되어 내부 DB에 저장된 통합 정책 데이터를 기준으로 동작한다.

```text
정책 수집 기능
→ 온통청년 데이터 수집
→ 행정안전부 공공서비스 정보 수집
→ 지자체 복지서비스 수집
→ 중앙부처 복지서비스 수집
→ 내부 DB 저장
→ 정책 데이터 정제
→ 통합 정책 데이터 생성

정책 검색 기능
→ 사용자 입력 조건 추출
→ 내부 DB에서 통합 정책 데이터 검색
→ 신청 가능성 판별
→ 결과 표시
```

이 구조를 사용하면 특정 외부 API 장애가 발생하더라도 이미 저장된 정책 데이터로 검색 기능을 유지할 수 있다.

## 21. 구현 시 주의사항

### 21.1 API 인증 정보 보안

API 인증 정보는 절대 GitHub에 올리지 않는다.

환경변수, 별도 설정 파일, 서버 환경변수 등을 사용하여 관리한다.

### 21.2 외부 API 장애 대비

외부 API 호출 실패 시 사용자 검색 기능이 중단되지 않도록 한다.

검색 기능은 내부 DB를 기준으로 수행하고, 수집 실패는 로그로 남긴다.

### 21.3 출처별 장애 분리

특정 출처의 API 호출이 실패하더라도 다른 출처 수집은 계속 진행한다.

예를 들어 온통청년 수집이 실패하더라도 지자체 복지서비스와 중앙부처 복지서비스 수집은 계속 진행한다.

### 21.4 데이터 정합성 유지

정책 데이터가 중복 저장되지 않도록 출처별 정책 식별값, 정책명, 기관명, 지역, 신청 기간 등을 기준으로 중복을 확인한다.

### 21.5 신청 기간 누락 대응

정책에 신청 기간이 없는 경우 캘린더에 표시하지 않거나 `확인 필요` 상태로 처리한다.

### 21.6 원본 데이터 보존

정제 과정에서 누락이나 오류가 발생할 수 있으므로 가능하면 원본 API 응답도 함께 저장한다.

### 21.7 자동 병합 주의

여러 출처에서 비슷한 이름의 정책이 수집되더라도 자동으로 하나의 정책으로 병합하지 않는다.

정책명은 비슷해도 지역, 기관, 기간, 대상이 다르면 다른 정책일 수 있으므로 자동 병합은 보수적으로 처리한다.

### 21.8 검색 기능과 책임 분리

수집 기능은 정책 데이터를 가져오고 정제하는 역할만 담당한다.

사용자의 자연어 입력을 분석하거나 신청 가능성을 판별하는 로직은 검색/추천 기능에서 처리한다.

## 22. 최종 정리

정책 데이터 수집 기능은 정책 추천 기능의 기반이 되는 핵심 기능이다.

사용자가 자연어로 정책을 검색하더라도, 실제 검색과 추천은 내부 DB에 저장된 통합 정책 데이터를 기준으로 수행한다.

따라서 여러 외부 출처에서 정책 데이터를 가져오고, 서비스에 맞게 정제하고, 중복을 처리하고, 주기적으로 갱신하는 구조가 필요하다.

최종 구조는 다음과 같다.

```text
온통청년
행정안전부 공공서비스 정보
지자체 복지서비스
중앙부처 복지서비스
↓
출처별 정책 데이터 수집
↓
원본 데이터 저장
↓
공통 정책 구조로 정제
↓
중복 확인 및 갱신
↓
통합 정책 데이터 생성
↓
정책 검색/추천
↓
신청 가능성 판별
↓
캘린더/알림 연동
```

초기 구현에서는 4개 출처를 기준으로 정책 데이터를 수집하고, 내부 DB에 저장한 뒤 자연어 검색 및 추천 기능에서 활용하는 방식으로 구현한다.
