# 정책 검색 및 추천 기능 설계서

## 1. 기능 개요

정책 검색 및 추천 기능은 사용자가 자연어로 입력한 문장을 기반으로, 사용자의 상황에 맞는 정부지원정책, 청년정책, 지역 복지정책, 중앙부처 복지서비스, 공공서비스 등을 추천하는 기능이다.

사용자는 복잡한 조건을 직접 필터로 선택하지 않고, 검색창에 다음과 같이 자유롭게 입력할 수 있다.

```text
경기도 사는 무직 청년이 받을 수 있는 지원금 알려줘
수원 청년 월세 지원 정책 있어?
대학생인데 받을 수 있는 정책 알려줘
나 받을 수 있는 거 있어?
소득 없는 취준생이 받을 수 있는 복지 알려줘
```

시스템은 사용자의 자연어 입력에서 지역, 나이/대상, 취업 상태, 학생 여부, 소득 조건, 주거 조건, 관심 분야 등의 조건을 추출한다.

그 후 내부 DB에 저장된 통합 정책 데이터와 비교하여 신청 가능성이 높은 정책을 추천한다.

정책은 신청 조건이 복잡하고 정책마다 요구하는 정보가 다르기 때문에, 조건이 부족한 경우에는 검색을 막지 않고 `확인 필요` 상태로 결과를 제공한다.

이후 사용자가 부족한 조건을 추가로 입력하거나 선택하면 신청 가능성을 다시 판별한다.

---

## 2. 기능 목표

### 2.1 사용자 목표

사용자는 자신이 받을 수 있는 정책을 복잡한 필터 조작 없이 자연어로 쉽게 검색할 수 있어야 한다.

예를 들어 사용자는 다음처럼 입력할 수 있다.

```text
내가 받을 수 있는 지원금 있어?
수원 사는 청년인데 월세 지원 받을 수 있어?
무직인데 신청 가능한 정책 알려줘
```

서비스는 사용자의 문장을 분석하여 정책 검색에 필요한 조건을 추출하고, 조건이 부족한 경우에는 자연스럽게 추가 정보를 요청한다.

---

### 2.2 서비스 목표

서비스는 사용자의 자연어 입력을 분석하여 정책 검색에 필요한 조건으로 변환한다.

이후 정책별 신청 조건과 비교하여 다음 정보를 제공한다.

- 추천 정책 목록
- 신청 가능성
- 신청 상태
- 추천 사유
- 부족한 정보
- 정책 상세 정보
- 관심 정책 저장
- 캘린더 연동

---

### 2.3 핵심 방향

이 기능의 핵심은 단순 키워드 검색이 아니다.

자연어 입력을 기반으로 사용자의 조건을 추출하고, 정책 조건과 비교해 신청 가능성을 판단하는 것이 핵심이다.

사용자 화면에서는 자연어 검색처럼 보이지만 내부적으로는 다음과 같은 구조로 동작한다.

```text
자연어 입력
↓
조건 추출
↓
부족한 조건 확인
↓
내부 DB의 통합 정책 데이터 검색
↓
신청 가능성 판별
↓
결과 표시
↓
부족 조건 보완
↓
신청 가능성 재판별
```

---

## 3. 정책 데이터 의존성

정책 검색 기능은 외부 API를 직접 호출하지 않는다.

정책 수집 기능을 통해 내부 DB에 저장된 통합 정책 데이터를 기준으로 동작한다.

정책 데이터의 수집, 정제, 중복 처리, 갱신 방식은 `policy_collection.md`에서 별도로 정의한다.

검색 기능이 사용하는 정책 데이터 출처는 다음과 같다.

| 데이터 출처                | 검색 활용 목적                    |
| -------------------------- | --------------------------------- |
| 온통청년                   | 청년정책 검색 및 추천             |
| 행정안전부 공공서비스 정보 | 공공서비스성 정책 검색 및 추천    |
| 지자체 복지서비스          | 지역 단위 복지서비스 검색 및 추천 |
| 중앙부처 복지서비스        | 중앙정부 복지서비스 검색 및 추천  |

검색 기능은 위 네 개 출처를 각각 직접 조회하지 않는다.

정책 수집 기능이 네 개 출처의 데이터를 수집하고, 내부 공통 정책 구조로 정제한 뒤 DB에 저장하면, 검색 기능은 해당 DB 데이터를 기준으로 검색과 추천을 수행한다.

```text
정책 수집 기능
→ 4개 출처에서 정책 데이터 수집
→ 내부 DB 저장
→ 정책 데이터 정제
→ 통합 정책 데이터 생성

정책 검색 기능
→ 사용자 자연어 입력 분석
→ 조건 추출
→ 내부 DB의 통합 정책 데이터 검색
→ 신청 가능성 판별
→ 결과 표시
```

이 구조를 사용하면 특정 외부 API 장애가 발생하더라도 이미 수집된 정책 데이터를 기준으로 검색 기능을 유지할 수 있다.

또한 검색 기능은 외부 API 응답 형식에 직접 의존하지 않으므로, 향후 새로운 정책 데이터 출처가 추가되어도 검색 로직을 크게 변경하지 않고 확장할 수 있다.

## 4. 핵심 설계 방향

### 4.1 자연어 검색 UI 제공

사용자에게는 별도의 복잡한 필터를 먼저 요구하지 않는다.

검색창 하나를 중심으로 자연어 입력을 받을 수 있도록 설계한다.

예시 문구는 다음과 같다.

```text
나에게 맞는 정책을 자유롭게 입력해보세요.
예: 수원 사는 무직 청년이 받을 수 있는 지원금 알려줘
```

사용자는 검색어를 문장 형태로 입력할 수 있다.

```text
나 받을 수 있는 거 있어?
수원 사는 청년인데 주거 지원 알려줘
대학생이 받을 수 있는 지원금 있어?
```

---

### 4.2 내부적으로는 조건 기반 검색 처리

사용자가 입력한 자연어 문장은 그대로 검색에 사용하지 않는다.

내부적으로 정책 검색 조건으로 변환한다.

예시 입력:

```text
경기도 사는 무직 청년이 받을 수 있는 지원금 알려줘
```

추출 결과 예시:

```json
{
  "region": "경기도",
  "ageGroup": "청년",
  "employmentStatus": "미취업",
  "studentStatus": null,
  "incomeLevel": null,
  "housingStatus": null,
  "keyword": "지원금"
}
```

이렇게 추출된 조건을 내부 DB에 저장된 통합 정책 데이터의 지원 대상, 지역, 신청 조건, 지원 분야 등과 비교하여 검색 결과를 제공한다.

---

### 4.3 검색 기능과 수집 기능 분리

검색 기능은 사용자의 입력을 분석하고, 내부 DB의 정책 데이터와 비교하여 결과를 제공하는 역할만 담당한다.

외부 API 호출, 정책 데이터 수집, 원본 데이터 저장, 데이터 정제, 중복 처리, 자동 갱신은 검색 기능에서 직접 처리하지 않는다.

해당 내용은 `policy_collection.md`에서 별도로 정의한다.

---

## 5. 자연어 입력에서 추출할 조건

자연어 검색 기능에서는 다음 조건을 우선적으로 추출한다.

| 조건      | 설명                               | 예시                                  |
| --------- | ---------------------------------- | ------------------------------------- |
| 지역      | 사용자의 거주 지역 또는 관심 지역  | 경기도, 수원시, 서울                  |
| 대상      | 정책 대상 계층                     | 청년, 대학생, 신혼부부, 구직자        |
| 나이      | 사용자의 실제 나이 또는 연령대     | 만 19세, 27살, 청년                   |
| 취업 상태 | 현재 경제활동 상태                 | 재직 중, 무직, 구직 중, 프리랜서      |
| 학생 여부 | 대학생, 졸업생, 휴학생 등          | 대학생, 취준생, 졸업예정자            |
| 소득 조건 | 소득 수준 또는 소득 유무           | 소득 없음, 중위소득 100% 이하         |
| 주거 조건 | 월세, 전세, 무주택 여부 등         | 월세 거주, 무주택                     |
| 가구 조건 | 1인 가구, 신혼부부, 한부모 가구 등 | 1인 가구, 신혼부부                    |
| 관심 분야 | 사용자가 찾는 정책 분야            | 주거, 취업, 생활비, 금융, 교육        |
| 키워드    | 정책 검색에 사용할 핵심 단어       | 지원금, 월세, 적금, 교육비            |
| 신청 상태 | 신청 가능한 정책 여부              | 지금 신청 가능한 정책, 마감 임박 정책 |

---

## 6. 조건 부족 상황 처리 방식

정책 검색 기능에서는 사용자가 항상 충분한 조건을 입력한다고 가정하지 않는다.

예를 들어 사용자는 다음과 같이 입력할 수 있다.

```text
나 받을 수 있는 정책 있어?
```

이 경우 지역, 나이, 취업 상태, 소득 조건, 관심 분야 등이 모두 부족하다.

조건이 부족한 경우에도 검색을 실패로 처리하지 않는다.

대신 조건 부족 정도에 따라 다음과 같이 처리한다.

- 조건이 충분하면 바로 검색한다.
- 조건이 일부 부족하면 결과를 먼저 보여주고 `확인 필요`로 표시한다.
- 조건이 대부분 부족하면 추가 질문을 먼저 제공한다.
- 특정 정책에서만 조건이 부족하면 해당 정책 카드에서 추가 질문을 제공한다.

---

## 7. 조건 부족 정도에 따른 처리 기준

### 7.1 조건이 충분한 경우

예시 입력:

```text
수원 사는 27살 무직 청년이 받을 수 있는 생활비 지원금 알려줘
```

추출 가능한 조건:

```text
지역: 수원시
나이: 27세
대상: 청년
취업 상태: 무직
관심 분야: 생활비 / 지원금
```

처리 방식은 다음과 같다.

```text
1. 정책 검색을 바로 진행한다.
2. 정책별 신청 조건과 사용자 조건을 비교한다.
3. 신청 가능성이 높은 정책을 우선 노출한다.
4. 결과 카드에 신청 가능성 상태를 표시한다.
```

화면 표시 예시:

```text
수원시 거주 청년 대상 정책을 찾았어요.
입력하신 조건을 기준으로 신청 가능성이 높은 정책부터 보여드릴게요.
```

---

### 7.2 조건이 일부 부족한 경우

예시 입력:

```text
수원 청년 지원금 알려줘
```

추출 가능한 조건:

```text
지역: 수원시
대상: 청년
관심 분야: 지원금
```

부족한 조건:

```text
나이
취업 상태
소득 조건
학생 여부
```

처리 방식은 다음과 같다.

```text
1. 검색을 막지 않는다.
2. 입력된 조건을 기준으로 정책 결과를 먼저 보여준다.
3. 신청 가능성은 확정하지 않고 `확인 필요`로 표시한다.
4. 부족한 조건을 사용자에게 안내한다.
```

화면 표시 예시:

```text
수원시 청년 대상 지원 정책을 찾았어요.
다만 나이, 소득, 취업 상태에 따라 신청 가능 여부가 달라질 수 있어요.
```

정책 카드 표시 예시:

```text
청년 월세 지원
신청 가능성: 확인 필요
부족한 정보: 소득 조건, 주거 형태, 무주택 여부
```

---

### 7.3 조건이 대부분 부족한 경우

예시 입력:

```text
나 받을 수 있는 거 있어?
```

추출 가능한 조건:

```text
없음 또는 매우 부족
```

처리 방식은 다음과 같다.

```text
1. 바로 정책 결과를 노출하기보다 추가 질문을 먼저 제공한다.
2. 사용자가 이탈하지 않도록 선택형 버튼 또는 간단한 보완 폼을 제공한다.
3. 사용자가 조건을 입력하면 그 조건을 기반으로 정책 검색을 진행한다.
```

화면 표시 예시:

```text
정확한 정책 추천을 위해 몇 가지 정보가 필요해요.
아래 항목을 선택하면 받을 수 있는 정책을 더 정확하게 찾아드릴게요.
```

선택 항목 예시:

```text
거주 지역
[경기도] [서울시] [수원시] [직접 입력]

현재 상태
[재직 중] [구직 중] [무직] [대학생] [프리랜서]

관심 분야
[주거] [취업] [생활비] [금융] [교육]
```

---

### 7.4 특정 정책에서만 조건이 부족한 경우

정책마다 필요한 조건은 다르다.

예를 들어 월세 지원 정책은 주거 형태와 무주택 여부가 중요하고, 취업 지원 정책은 구직 여부와 졸업 여부가 중요할 수 있다.

따라서 모든 정책에 동일한 추가 질문을 적용하지 않는다.

검색 결과를 먼저 보여준 뒤, 각 정책 카드에서 해당 정책에 필요한 부족 조건만 안내한다.

예시:

```text
청년 월세 지원
신청 가능성: 확인 필요

이 정책은 주거 형태 확인이 필요해요.
현재 월세 거주 중인가요?

[예] [아니오] [잘 모르겠음]
```

## 8. 신청 가능성 상태

정책 추천 결과는 단순히 신청 가능 / 불가능으로만 나누지 않는다.

조건 부족이나 애매한 상황이 많기 때문에 다음과 같은 상태를 사용한다.

| 상태        | 설명                                              |
| ----------- | ------------------------------------------------- |
| 신청 가능   | 현재 입력된 조건 기준으로 신청 가능성이 높음      |
| 가능성 높음 | 주요 조건은 충족하지만 일부 세부 확인이 필요함    |
| 확인 필요   | 신청 여부 판단에 필요한 조건이 부족함             |
| 신청 어려움 | 현재 조건 기준으로 주요 신청 조건을 충족하지 못함 |
| 판별 불가   | 정책 조건이 명확하지 않거나 데이터가 부족함       |

조건이 완전히 확인되지 않은 상태에서는 `신청 가능`이라고 단정하지 않는다.

---

## 9. 정책 신청 상태

정책 자체의 신청 기간 상태는 다음과 같이 표시한다.

| 상태      | 설명                     |
| --------- | ------------------------ |
| 신청 가능 | 현재 신청 기간에 해당함  |
| 신청 예정 | 아직 신청 시작 전        |
| 신청 마감 | 신청 마감일이 지남       |
| 상시 신청 | 상시 신청 가능한 정책    |
| 확인 필요 | 신청 기간 정보가 부족함  |
| 비활성    | 사용자에게 노출하지 않음 |

신청 가능성 상태와 정책 신청 상태는 구분한다.

예를 들어 사용자의 조건에는 맞지만 신청 기간이 끝났다면 다음처럼 표시할 수 있다.

```text
신청 가능성: 가능성 높음
신청 상태: 신청 마감
```

## 10. 조건 부족 시 사용자 안내 기준

조건이 부족할 때는 사용자가 실패했다고 느끼지 않도록 안내해야 한다.

나쁜 예시는 다음과 같다.

```text
조건이 부족합니다. 다시 입력해주세요.
```

좋은 예시는 다음과 같다.

```text
입력하신 내용만으로도 관련 정책을 찾아봤어요.
다만 정확한 신청 가능 여부를 확인하려면 몇 가지 정보가 더 필요해요.
```

또는 다음처럼 안내할 수 있다.

```text
수원시 청년 대상 정책을 먼저 보여드릴게요.
소득, 취업 상태, 주거 형태를 추가하면 더 정확하게 추천할 수 있어요.
```

## 11. 정책 검색 결과 카드 구성

정책 검색 결과는 카드 형태로 제공한다.

카드에는 다음 정보가 포함될 수 있다.

| 항목           | 설명                                                              |
| -------------- | ----------------------------------------------------------------- |
| 정책명         | 정책 이름                                                         |
| 주관 기관      | 정책을 운영하는 기관                                              |
| 데이터 출처    | 온통청년, 공공서비스 정보, 지자체 복지서비스, 중앙부처 복지서비스 |
| 지원 대상      | 청년, 구직자, 대학생 등                                           |
| 지원 내용      | 지원금, 월세 지원, 교육비 지원 등                                 |
| 신청 기간      | 신청 시작일, 마감일                                               |
| 신청 상태      | 신청 가능, 신청 예정, 신청 마감, 상시 신청                        |
| 신청 가능성    | 신청 가능, 가능성 높음, 확인 필요, 신청 어려움                    |
| 부족한 정보    | 신청 가능성 판단에 필요한 추가 정보                               |
| 추천 사유      | 왜 이 정책이 추천되었는지 설명                                    |
| 상세 보기      | 정책 상세 페이지 이동                                             |
| 관심 정책 저장 | 캘린더 및 알림 연동을 위한 저장 기능                              |

카드 예시는 다음과 같다.

```text
청년 월세 지원

출처: 온통청년
주관 기관: 수원시
지원 대상: 수원시 거주 청년
지원 내용: 월세 일부 지원
신청 기간: 2026.07.01 ~ 2026.07.31
신청 상태: 신청 가능
신청 가능성: 확인 필요

추천 사유:
입력하신 지역과 청년 조건에 맞는 정책이에요.

부족한 정보:
소득 조건, 주거 형태, 무주택 여부 확인이 필요해요.

[상세 보기] [관심 정책 저장]
```

## 12. 여러 출처의 중복 정책 표시 기준

네 개 출처에서 정책 데이터를 수집하기 때문에 같은 정책 또는 유사한 정책이 여러 출처에서 발견될 수 있다.

검색 결과에서는 중복 정책을 무조건 여러 개로 보여주지 않는다.

수집 기능에서 병합 또는 그룹화된 결과를 기준으로 표시한다.

### 12.1 동일 정책으로 병합된 경우

같은 정책으로 판단된 경우 하나의 카드로 표시하고, 출처를 함께 보여준다.

```text
청년 월세 지원

출처: 온통청년, 지자체 복지서비스
주관 기관: 수원시
신청 가능성: 확인 필요
```

---

### 12.2 별도 정책으로 유지된 경우

정책명이 비슷하더라도 지역, 기관, 신청 기간, 지원 내용이 다르면 별도 정책으로 표시한다.

예시:

```text
청년 월세 지원 - 경기도
청년 월세 지원 - 수원시
```

중복 판단과 병합 기준은 `policy_collection.md`에서 정의한다.

## 13. 자연어 검색 처리 흐름

기본 검색 흐름은 다음과 같다.

```text
1. 사용자가 자연어 검색어 입력
2. 검색어에서 조건 추출
3. 추출된 조건을 정책 검색 조건으로 변환
4. 조건 부족 여부 확인
5. 내부 DB의 통합 정책 데이터 검색
6. 정책별 신청 조건 비교
7. 신청 가능성 상태 계산
8. 정책 신청 상태 확인
9. 결과 카드 표시
10. 부족 조건 안내
11. 사용자가 조건 보완 시 신청 가능성 재판별
```

검색 대상은 외부 API가 아니라 내부 DB에 저장된 통합 정책 데이터다.

검색 대상 데이터는 다음 출처에서 수집된 정책을 포함한다.

```text
온통청년
행정안전부 공공서비스 정보
지자체 복지서비스
중앙부처 복지서비스
```

## 14. 정책 추천 판별 기준

정책 추천 판별은 다음 기준을 종합하여 처리한다.

| 판별 기준      | 설명                                            |
| -------------- | ----------------------------------------------- |
| 지역 일치 여부 | 사용자의 지역과 정책 대상 지역 비교             |
| 연령 조건      | 사용자의 나이 또는 연령대와 정책 연령 조건 비교 |
| 대상 조건      | 청년, 대학생, 구직자 등 대상 조건 비교          |
| 취업 상태      | 재직, 미취업, 구직 중 여부 비교                 |
| 학생 여부      | 대학생, 졸업생, 휴학생 등 여부 비교             |
| 소득 조건      | 소득 기준 충족 여부 비교                        |
| 주거 조건      | 무주택, 월세, 전세 등 조건 비교                 |
| 가구 조건      | 1인 가구, 신혼부부, 한부모 등 조건 비교         |
| 신청 기간      | 현재 신청 가능한 정책인지 확인                  |
| 관심 분야      | 사용자가 찾는 분야와 정책 분야 비교             |
| 데이터 출처    | 동일 정책이 여러 출처에서 확인되는지 여부       |

## 15. 추가 질문 및 조건 보완 기능

조건이 부족한 경우, 사용자가 자연어를 다시 입력하지 않아도 쉽게 조건을 보완할 수 있도록 선택형 UI를 제공한다.

### 15.1 공통 추가 질문

| 질문                              | 선택 예시                                |
| --------------------------------- | ---------------------------------------- |
| 거주 지역이 어디인가요?           | 경기도, 서울시, 수원시, 직접 입력        |
| 현재 상태는 어떤가요?             | 재직 중, 구직 중, 무직, 대학생, 프리랜서 |
| 관심 있는 정책 분야는 무엇인가요? | 주거, 취업, 생활비, 금융, 교육           |
| 소득이 있나요?                    | 있음, 없음, 잘 모르겠음                  |
| 학생인가요?                       | 대학생, 졸업생, 휴학생, 해당 없음        |
| 주거 형태는 어떤가요?             | 월세, 전세, 자가, 해당 없음              |
| 가구 형태는 어떤가요?             | 1인 가구, 신혼부부, 한부모, 해당 없음    |

---

### 15.2 정책별 추가 질문

정책마다 필요한 조건이 다르기 때문에 상세 조건은 정책 카드 또는 상세 페이지에서 개별적으로 질문한다.

예시:

```text
이 정책은 월세 거주 여부 확인이 필요해요.
현재 월세로 거주 중인가요?

[예] [아니오] [잘 모르겠음]
```

```text
이 정책은 중소기업 재직 여부 확인이 필요해요.
현재 중소기업에 재직 중인가요?

[예] [아니오] [잘 모르겠음]
```

```text
이 정책은 기준 중위소득 조건 확인이 필요해요.
가구 소득 기준을 알고 있나요?

[예] [아니오] [잘 모르겠음]
```

## 16. 구현 범위

초기 구현에서는 자연어 검색의 모든 예외를 완벽하게 처리하기보다, 핵심 흐름을 먼저 구현한다.

### 16.1 1차 구현 범위

```text
- 자연어 검색창 제공
- 지역, 대상, 관심 분야 중심의 조건 추출
- 내부 DB의 통합 정책 목록 조회
- 정책 상세 조회
- 신청 가능성 상태 표시
- 정책 신청 상태 표시
- 조건 부족 시 `확인 필요` 표시
- 부족한 정보 안내
- 정책 출처 표시
- 관심 정책 저장
- 관심 정책 캘린더 표시
```

---

### 16.2 2차 구현 범위

```text
- 취업 상태, 소득, 학생 여부 등 추가 조건 추출
- 조건 보완 버튼 또는 폼 제공
- 조건 보완 후 신청 가능성 재판별
- 정책별 추가 질문 제공
- 추천 사유 고도화
- 중복 정책 결과 그룹화
- 출처별 필터 제공
```

---

### 16.3 확장 구현 범위

```text
- RAG 기반 정책 상세 설명 요약
- 정책 신청 방법 요약
- 정책별 필요 서류 안내
- 수집된 정책 데이터 기반 검색 품질 고도화
- 정책 마감 알림
- 사용자 프로필 기반 자동 추천
- 정책 출처별 최신성 표시
```

정책 데이터 수집 및 갱신 자체는 `policy_collection.md`에서 관리한다.

## 17. 예외 상황 및 처리 방식

| 예외 상황             | 예시                         | 처리 방식                                  |
| --------------------- | ---------------------------- | ------------------------------------------ |
| 조건이 거의 없음      | 나 받을 수 있는 거 있어?     | 추가 질문 먼저 제공                        |
| 지역만 있음           | 수원 정책 알려줘             | 지역 기반 정책 표시 후 대상 조건 보완 요청 |
| 대상만 있음           | 청년 정책 알려줘             | 청년 정책 표시 후 지역 조건 보완 요청      |
| 의미가 애매함         | 학생 지원금 알려줘           | 대학생/고등학생/취준생 등 후보 선택 제공   |
| 정책 조건 부족        | 소득 기준 데이터 없음        | 판별 불가 또는 확인 필요 표시              |
| 신청 기간 종료        | 마감된 정책                  | 신청 어려움 또는 마감 표시                 |
| 검색 결과 없음        | 조건에 맞는 정책 없음        | 조건 완화 검색 또는 인기 정책 추천         |
| 여러 지역 후보 존재   | 광주 정책 알려줘             | 광주광역시/경기도 광주시 선택 요청         |
| 중복 정책 존재        | 같은 정책이 여러 출처에 있음 | 하나의 카드로 묶거나 출처 목록 표시        |
| 특정 출처 데이터 누락 | 일부 API 수집 실패           | 내부 DB의 기존 데이터 기준으로 검색 유지   |
| 신청 기간 정보 없음   | 기간 미제공 정책             | 신청 상태를 확인 필요로 표시               |

## 18. 사용자 화면 흐름 예시

### 18.1 조건이 충분한 검색

```text
사용자 입력:
수원 사는 무직 청년이 받을 수 있는 생활비 지원금 알려줘

시스템 응답:
수원시 거주 청년 및 미취업자를 대상으로 하는 생활비 관련 정책을 찾았어요.
입력하신 조건을 기준으로 신청 가능성이 높은 정책부터 보여드릴게요.
```

결과 카드:

```text
청년 생활안정 지원금

출처: 지자체 복지서비스
신청 가능성: 가능성 높음
신청 상태: 신청 가능
추천 사유: 수원시, 청년, 미취업 조건과 관련된 정책이에요.
부족한 정보: 소득 조건 확인 필요
```

---

### 18.2 조건이 일부 부족한 검색

```text
사용자 입력:
수원 청년 지원금 알려줘

시스템 응답:
수원시 청년 대상 지원 정책을 찾았어요.
다만 소득, 취업 상태, 학생 여부에 따라 신청 가능 여부가 달라질 수 있어요.
```

결과 카드:

```text
청년 월세 지원

출처: 온통청년, 지자체 복지서비스
신청 가능성: 확인 필요
신청 상태: 신청 가능
부족한 정보: 소득 조건, 주거 형태, 무주택 여부
```

사용자 입력:

선택 UI:

```text

현재 상태: [재직 중] [구직 중] [무직] [대학생] [프리랜서]
관심 분야: [주거] [취업] [생활비] [금융] [교육]
```

## 19. 데이터 구조 예시

아래 정책 데이터는 외부 API에서 직접 받은 원본 데이터가 아니라, 수집 기능을 통해 내부 DB에 저장되고 정제된 통합 정책 데이터를 기준으로 한다.

외부 API 응답 원문, 수집 로그, 중복 처리, 데이터 갱신 방식은 `policy_collection.md`에서 정의한다.

## 21. 캘린더 연동 기준

사용자가 관심 정책으로 저장한 정책은 캘린더에 표시한다.

캘린더 표시 대상은 다음과 같다.

```text
- 관심 정책으로 저장한 정책
- 신청 시작일이 있는 정책
- 신청 마감일이 있는 정책
- 상시 신청이 아닌 정책
```

캘린더 표시 방식은 다음과 같다.

| 일정 종류   | 표시 내용                       |
| ----------- | ------------------------------- |
| 신청 시작일 | 정책 신청 시작                  |
| 신청 마감일 | 정책 신청 마감                  |
| 신청 기간   | 시작일부터 마감일까지 기간 표시 |

캘린더 예시는 다음과 같다.

```text
2026-07-01
청년 월세 지원 신청 시작

2026-07-31
청년 월세 지원 신청 마감
```

## 23. 구현 시 주의사항

### 23.1 자연어 검색 결과를 확정적으로 말하지 않기

정책 조건은 복잡하고 사용자의 정보가 부족할 수 있다.

따라서 조건이 완전히 충족되지 않은 상태에서는 `신청 가능`이라고 단정하지 않는다.

대신 다음과 같이 표현한다.

```text
신청 가능
가능성 높음
확인 필요
신청 어려움
판별 불가
```

---

### 23.2 검색 실패로 처리하지 않기

조건이 부족하다는 이유로 검색을 실패 처리하지 않는다.

가능한 경우 관련 정책을 먼저 보여주고, 부족한 정보를 안내한다.

---

### 23.3 사용자에게 입력 부담을 주지 않기

처음부터 긴 설문지를 제공하지 않는다.

자연어 검색을 먼저 받고, 필요한 경우에만 추가 조건을 요청한다.

---

### 23.4 정책별 조건 차이를 고려하기

모든 정책에 동일한 조건을 요구하지 않는다.

정책별로 필요한 조건이 다르므로, 정책 카드 또는 상세 페이지에서 필요한 조건만 추가로 확인한다.

---

### 23.5 정책 수집 로직과 검색 로직 분리

검색 기능은 사용자의 자연어 입력을 분석하고, 내부 DB의 정책 데이터와 비교하여 신청 가능성을 판별하는 역할을 담당한다.

외부 API 호출, 정책 데이터 수집, 원본 데이터 저장, 정책 데이터 정제, 중복 처리, 자동 갱신은 검색 기능에서 직접 처리하지 않는다.

해당 내용은 `policy_collection.md`에서 별도로 정의한다.

---

### 23.6 출처별 데이터 차이를 고려하기

같은 정책이라도 출처별로 제공하는 정보의 양과 표현 방식이 다를 수 있다.

검색 기능은 수집 기능에서 정제된 공통 정책 데이터를 기준으로 동작하되, 상세 페이지에서는 필요한 경우 출처 정보를 함께 표시한다.

---

### 23.7 중복 정책 노출 주의

여러 출처에서 같은 정책이 수집된 경우 검색 결과에 같은 정책이 반복 노출되지 않도록 한다.

중복 판단 및 병합 기준은 수집 기능에서 처리하고, 검색 기능은 병합된 정책 데이터를 기준으로 결과를 표시한다.

---

## 24. 최종 정리

정책 검색 및 추천 기능은 사용자가 자연어로 편하게 정책을 찾을 수 있도록 설계한다.

다만 내부적으로는 자연어 입력을 그대로 검색하는 것이 아니라, 지역, 대상, 취업 상태, 소득, 관심 분야 등의 조건으로 변환한 뒤 내부 DB에 저장된 통합 정책 데이터와 비교한다.

조건이 부족한 경우에는 검색을 막지 않고 `확인 필요` 상태로 결과를 제공한다.

이후 사용자가 부족한 조건을 선택하거나 입력하면 신청 가능성을 다시 판별한다.

정책 검색 기능은 정책 수집 기능에 의해 내부 DB에 저장된 정책 데이터를 기준으로 동작한다.

따라서 검색 기능의 책임은 외부 정책 API를 호출하는 것이 아니라, 사용자의 자연어 입력을 조건으로 변환하고 내부 정책 데이터와 비교하여 신청 가능성을 판별하는 것이다.

정책 데이터 수집, 정제, 갱신, 실패 처리 방식은 `policy_collection.md`에서 별도로 관리한다.

최종 구조는 다음과 같다.

```text
겉으로는 자연어 검색
속으로는 조건 추출 + 통합 정책 데이터 검색
결과에서는 신청 가능성 단계 표시
부족한 조건은 추가 질문으로 보완
보완 후 신청 가능성 재판별
```

이 방식은 자연어 검색의 편의성을 유지하면서도, 여러 출처에서 수집된 정책 데이터를 안정적으로 활용할 수 있게 한다.
